## Возможности

- Стабилизация дрожания камеры (`vid.stab`) с полным контролем параметров
  обоих проходов — точности анализа и последующего сглаживания/зума.
- Опциональное шумоподавление (Smart Blur) и повышение резкости после
  стабилизации (unsharp и/или Contrast Adaptive Sharpening).
- Компрессия видео `libx264` в двух режимах: постоянное качество (CRF,
  1 проход) либо точный целевой битрейт (2 прохода).
- Копирование звука, субтитров и вложений контейнера без перекодирования.
- Прогресс обработки и статус — прямо в интерфейсе, без блокировки окна.
- Переключение языка интерфейса (русский/английский) без перезапуска.

## Вкладки приложения

### 1. Файл

![Вкладка «Файл»](../images/Screenshot_1.png)

- Выбор входного видео.
- Выбор пути сохранения результата.
- Выбор контейнера результата — **mp4** либо **mkv**. Это определяет
  расширение файла и, соответственно, мьюксер libavformat: mkv надёжнее
  сохраняет субтитры и вложения, mp4 совместимее с проигрывателями и
  сайтами. Расширение файла в диалоге «Куда сохранить...» подставляется
  автоматически по этому выбору.
- Независимые переключатели:
  - «Копировать звук без перекодирования»
  - «Сохранить субтитры»
  - «Сохранить вложения» (шрифты и т. п. в mkv)

  Все три включены по умолчанию.

### 2. Стабилизация

![Вкладка «Стабилизация»](../images/Screenshot_2.png)

**Проход 1 (`vidstabdetect`)** — анализ движения:

- `shakiness` (1–10)
- `accuracy` (1–15)
- `stepsize`
- `mincontrast`

По умолчанию параметры выставлены на максимум точности анализа
(10 / 15 / 6 / 0.3).

**Проход 2 (`vidstabtransform`)** — компенсация:

- `smoothing` — сглаживание траектории по кадрам
- `zoom` — базовый зум, %
- `optzoom` — автозум: 0 = выкл, 1 = статичный, 2 = адаптивный
- `crop` — обработка краёв кадра:
  - `keep` — дотянуть картинку с предыдущего кадра по краям
  - `black` — честные чёрные поля там, где после компенсации сдвига не
    хватает исходного изображения
- `interpol` — интерполяция при трансформации кадра (по умолчанию
  `bicubic` — самая точная)
- `maxshift` / `maxangle` — ограничения максимального сдвига (px) и
  угла поворота (рад); значение `-1` означает «без ограничения»

### 3. Резкость

![Вкладка «Резкость»](../images/Screenshot_3.png)

- **Smart Blur (`smartblur`)** — шумоподавление/смягчение
  (`luma_radius`, `luma_strength`, `luma_threshold`). Ставится ПЕРЕД
  фильтрами резкости ниже: резкость усиливает уже имеющийся шум,
  поэтому шумоподавление имеет смысл делать раньше.
- **Unsharp** (`luma_amount`) и **Contrast Adaptive Sharpening (CAS)**
  (`strength`) — применяются ПОСЛЕ стабилизации/зума, иначе усиливались
  бы артефакты исходной тряски, а не детали кадра. CAS меньше «звенит»
  на контрастных краях, чем классический unsharp. Оба фильтра резкости
  можно включить одновременно; Smart Blur — независимый переключатель.

### 4. Компрессия

![Вкладка «Компрессия»](../images/Screenshot_4.png)

- `preset` энкодера `libx264`.
- Режим управления битрейтом:
  - **1 проход (CRF)** — постоянное качество на всём ролике; быстрее,
    но итоговый размер файла заранее не предсказать. Для наивысшего
    качества — `preset=slow`/`slower` и `CRF 16–18`.
  - **2 прохода (задать битрейт)** — `libx264` сначала проходит весь
    ролик, считая статистику сложности сцен (проход A), затем кодирует
    повторно (проход B), распределяя биты так, чтобы точно попасть в
    заданный целевой битрейт видео. Вдвое медленнее, зато размер файла
    предсказуем.

  Ползунок «Целевой битрейт видео» активен только в режиме 2 проходов,
  CRF — только в режиме 1 прохода; переключение режима сразу блокирует
  неиспользуемый контрол.

## Почему GTK4 не статический

FFmpeg/x264/vidstab — библиотеки без рантайм-загрузки посторонних
модулей, поэтому их можно слинковать статически как обычный код. GTK4
устроен иначе: у него модульная система загрузчиков (gdk-pixbuf,
icon-тема, GSettings-схемы, immodules), которая ищет `.so`/`.dll` и
файлы данных по стандартным путям уже во время работы приложения — это
не решается на этапе линковки. Поэтому:

- **Fedora**: GTK4 ставится системно (`sudo dnf install gtk4-devel`
  для сборки, `gtk4` для запуска) — так делает подавляющее большинство
  GTK-приложений в дистрибутиве.
- **Windows**: рантайм GTK4 собирается через MSYS2/mingw64
  (`pacman -S mingw-w64-x86_64-gtk4`) и распространяется рядом с
  `Monolit.exe` (DLL + `share/glib-2.0/schemas` + `share/icons`) —
  ровно так же поступают GIMP, Inkscape и другие GTK-приложения на
  Windows. Полностью статический GTK никто в индустрии не поставляет.

## Почему оба прохода стабилизации идут последовательно, а не параллельно

`vidstabdetect` строит единую траекторию движения камеры по всему
ролику, а `vidstabtransform` сглаживает её с оглядкой на кадры и до, и
после текущего. Если бы видео резалось на независимые куски для
параллельной обработки в разных потоках, каждый кусок получил бы свою,
никак не согласованную с соседями траекторию — на стыках появились бы
заметные рывки, что свело бы на нет саму стабилизацию. Поэтому оба
прохода в Monolit идут по видео целиком и строго последовательно.

Параллелизм при этом никуда не делся — он остаётся на уровне
многопоточности самого декодера/энкодера/фильтра FFmpeg
(`thread_count`/`thread_type`), а вся обработка выполняется в отдельном
потоке ОС только для того, чтобы не блокировать событийный цикл GTK.

## Числовая локаль и FFmpeg (LC_NUMERIC)

GTK4 при инициализации сам вызывает `setlocale(LC_ALL, "")`, подхватывая
системную локаль пользователя целиком — в том числе `LC_NUMERIC`. В
локалях, где десятичный разделитель — запятая (`ru_RU`, `de_DE` и
большинство европейских), это ломает разбор числовых аргументов
фильтр-графов FFmpeg: `libavutil` (`av_expr`/`strtod`) при парсинге,
например, `mincontrast=0.300` в такой локали читает только `0`,
натыкается на `.300` и падает с ошибкой вида
`Invalid chars '.300' at the end of expression '0.300'`. То же самое
затронуло бы и `zoom`, `maxangle`, параметры `smartblur`, `unsharp`,
`cas`.

Nim форматирует float всегда через точку независимо от локали процесса,
так что сама генерация строки корректна — ломается именно её разбор
фильтром. `libavutil` всегда ожидает `C`-локаль для чисел, поэтому
Monolit после старта GTK (в обработчике `activate`, уже после того как
GTK сделал свой `setlocale`) принудительно возвращает `LC_NUMERIC` в
`"C"`, не трогая остальные категории локали (переводы интерфейса и т. п.
не страдают).

## Сборка: подробности

```bash
# Fedora, нативно
nim c -d:release --threads:on Monolit.nim

# Windows, кросс-компиляция с Fedora (mingw-w64)
nim c -d:release --threads:on --os:windows Monolit.nim
```

Первый запуск для каждой цели соберёт с нуля libvidstab → (libx264 на
Windows) → статический FFmpeg — это займёт время; повторные сборки
используют кэш в `vidstab_build*/`, `x264_build_windows/`,
`ffmpeg_build*/` (см. `config.nims`).

Перед сборкой под Windows на Fedora должен быть установлен рантайм GTK4
для mingw64 (`sudo dnf install mingw64-gtk4` либо через MSYS2), иначе
`pkg-config --libs gtk4` в `src/gtk4_api.nim` не найдёт нужные `.dll.a`.

Остальные зависимости (vid.stab, x264, FFmpeg) `config.nims` собирает
сам из исходников при первой сборке — вручную ставить их не нужно.

## Известная историческая проблема Windows-сборки (исправлено)

Первая кросс-сборка под Windows могла падать ещё на этапе сборки
libvidstab с ошибкой CMake вида `ld.bfd: неизвестный параметр
«--major-image-version»` и обрывом на `CMakeTestCXXCompiler`. Причина —
CMakeLists.txt самого vid.stab не ограничивает список языков явно и
включает C++ наравне с C, а `config.nims` указывал компилятор C
(`CMAKE_C_COMPILER`) для кросс-сборки, но не C++ — в результате CMake
находил хостовый (нативный) `c++`, при этом `CMAKE_SYSTEM_NAME=Windows`
всё равно заставлял его добавлять PE-специфичные флаги компоновщика,
которые нативный `ld.bfd` не понимает. Исправлено явным указанием
`CMAKE_CXX_COMPILER=x86_64-w64-mingw32-g++` (и заодно
`CMAKE_RC_COMPILER`) рядом с C-компилятором — см. подробный комментарий
в `buildVidstab` в `config.nims`.

После этого сборка идёт дальше (vidstab → dav1d → FFmpeg), но может
упасть ещё раз уже на этапе компиляции самих Nim-исходников с ошибкой
`pkg-config: Package gtk4 was not found in the pkg-config search path` —
причём текст этой ошибки от pkg-config, попав в `gorge()`, утекает прямо
в аргументы `gcc` (лог пестрит «linker input file not found» для слов
вроде `Package`/`was`/`not`/`found`).

Причина двойная: `PKG_CONFIG_LIBDIR` для Windows раньше настраивался
только внутри `buildFFmpeg` (и не выставлялся вовсе при закэшированной
FFmpeg-сборке), а сам список каталогов не включал стандартный путь
Fedora для .pc-файлов mingw64-пакетов.

Исправлено:

- настройка `PKG_CONFIG_LIBDIR`/`PATH` теперь выполняется безусловно
  при каждой сборке;
- список включает
  `/usr/x86_64-w64-mingw32/sys-root/mingw/lib/pkgconfig` (где Fedora
  кладёт `gtk4.pc` для `mingw64-gtk4`);
- `config.nims`/`src/gtk4_api.nim` дополнительно проверяют код возврата
  `pkg-config` и останавливают сборку понятным сообщением вместо того,
  чтобы передавать текст ошибки компилятору как флаги.

## Остановка обработки и закрытие окна

Кнопка «Стоп» просит фоновый поток остановиться на ближайшей проверке
между пакетами и дожидается его завершения асинхронно (через таймер
опроса прогресса), не блокируя интерфейс.

Закрытие окна во время активной обработки обрабатывается отдельно и
иначе: чтобы не оставить FFmpeg-контексты недомученными, а выходной
файл — недописанным и потенциально повреждённым, обработчик
`close-request` синхронно просит поток остановиться и дожидается его
завершения (ценой краткой паузы интерфейса) и только после этого
разрешает окну закрыться. Недописанный результат при этом удаляется
автоматически, как и при обычной остановке по кнопке.

## Стиль кода

Код проекта придерживается нескольких соглашений оформления:

- **Синтаксис вызова**: имя функции всегда идёт первым, все аргументы —
  в круглых скобках после него (`startsWith(str, "John")`, `len(a)`,
  `close(f)`), а не через точку (`str.startsWith("John")`). Точка
  используется только для обращения к полю структуры/объекта
  (`cfg.inputFile`, `w.progressBar`) — никогда для вызова процедур или
  преобразования типов (`cint(x)`, а не `x.cint`). Исключение — `echo`.
- **Объявление двух и более констант/переменных** одного вида идёт
  блоком: ключевое слово `const`/`let`/`var` на своей строке, а сами
  объявления — с отступом на следующих строках, а не по одному
  объявлению на `let`/`var` в строке.
- **Импорты** объединяются через запятую в одну строку (`import
  std/[strformat, os, strutils]`, `import gtk4_api, stabilizer`);
  несколько отдельных `import` оставляются только там, где они логически
  разделены (например, отдельный импорт с поясняющим комментарием только
  к нему).

## Язык интерфейса

Внизу вкладки «Файл» есть переключатель языка интерфейса (Русский /
English), переключается без перезапуска приложения. По умолчанию —
русский. Технические списки (mp4/mkv, keep/black,
no/linear/bilinear/bicubic, пресеты libx264) не переводятся
намеренно — это общепринятые технические обозначения, одинаковые в
обоих языках.

## Относительные пути в служебных файлах сборки

Сгенерированный `vidstab_build*/lib/pkgconfig/vidstab.pc` использует
переменную `${pcfiledir}` (её подставляет сам pkg-config — это каталог,
где лежит сам `.pc`-файл) вместо абсолютного пути каталога проекта,
зашитого на момент сборки. Это значит, что каталог проекта можно
переименовать, скопировать на другую машину или собрать под другим
пользователем — `.pc` остаётся рабочим без правки.
