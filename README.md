
# Monolit — GTK4-приложение для стабилизации видео (libvidstab)

Убирает дрожание камеры в видео с помощью двухпроходного алгоритма
`vid.stab` (`vidstabdetect` → `vidstabtransform`) поверх FFmpeg. FFmpeg,
libx264 и libvidstab компилируются в набор статических библиотек (`.a`)
и линкуются в бинарник напрямую — готовое приложение не тянет за собой
системные `.so`/`.dll` кодеков. GTK4 остаётся системной динамической
зависимостью — почему так, см. раздел «Почему GTK4 не статический» ниже.

## Возможности

- Стабилизация дрожания камеры (`vid.stab`) с полным контролем параметров
  обоих проходов — точности анализа и последующего сглаживания/зума.
- Опциональное шумоподавление (Smart Blur) и повышение резкости после
  стабилизации (unsharp и/или Contrast Adaptive Sharpening).
- Компрессия видео `libx264` в двух режимах: постоянное качество (CRF,
  1 проход) либо точный целевой битрейт (2 прохода).
- Копирование звука, субтитров и вложений контейнера без перекодирования.
- Прогресс обработки и статус — прямо в интерфейсе, без блокировки окна.

## Почему GTK4 не статический

FFmpeg/x264/vidstab — библиотеки без рантайм-загрузки посторонних
модулей, поэтому их можно слинковать статически как обычный код. GTK4
устроен иначе: у него модульная система загрузчиков (gdk-pixbuf,
icon-тема, GSettings-схемы, immodules), которая ищет `.so`/`.dll` и
файлы данных по стандартным путям УЖЕ ВО ВРЕМЯ РАБОТЫ приложения —
это не решается на этапе линковки. Поэтому:

- **Fedora**: GTK4 ставится системно (`sudo dnf install gtk4-devel`
  для сборки, `gtk4` для запуска) — так делает подавляющее большинство
  GTK-приложений в дистрибутиве.
- **Windows**: рантайм GTK4 собирается через MSYS2/mingw64
  (`pacman -S mingw-w64-x86_64-gtk4`) и распространяется РЯДОМ с
  `Monolit.exe` (DLL + `share/glib-2.0/schemas` + `share/icons`) —
  ровно так же поступают GIMP, Inkscape и другие GTK-приложения на
  Windows. Полностью статический GTK никто в индустрии не поставляет.

## Почему оба прохода стабилизации идут последовательно, а не параллельно

`vidstabdetect` строит ЕДИНУЮ траекторию движения камеры по всему
ролику, а `vidstabtransform` сглаживает её с оглядкой на кадры и до, и
после текущего. Если бы видео резалось на независимые куски для
параллельной обработки в разных потоках, каждый кусок получил бы свою,
никак не согласованную с соседями траекторию — на стыках появились бы
заметные рывки, что свело бы на нет саму стабилизацию. Поэтому оба
прохода в Monolit идут по видео целиком и строго последовательно.
Параллелизм при этом никуда не делся — он остаётся на уровне
многопоточности самого декодера/энкодера/фильтра FFmpeg
(`thread_count`/`thread_type`), а вся обработка выполняется в отдельном
потоке ОС только для того, чтобы не блокировать событийный цикл GTK.

## Числовая локаль и FFmpeg (LC_NUMERIC)

GTK4 при инициализации сам вызывает `setlocale(LC_ALL, "")`, подхватывая
системную локаль пользователя целиком — в том числе `LC_NUMERIC`. В
локалях, где десятичный разделитель — запятая (`ru_RU`, `de_DE` и
большинство европейских), это ломает разбор числовых аргументов
фильтр-графов FFmpeg: `libavutil` (`av_expr`/`strtod`) при парсинге,
например, `mincontrast=0.300` в такой локали читает только `0`,
натыкается на `.300` и падает с `Invalid chars '.300' at the end of
expression '0.300'` — то же самое затронуло бы и `zoom`, `maxangle`,
параметры `smartblur`, `unsharp`, `cas`. Nim форматирует float всегда
через точку независимо от локали процесса, так что сама генерация
строки корректна — ломается именно её разбор фильтром. `libavutil`
всегда ожидает `C`-локаль для чисел, поэтому Monolit после старта GTK
(в обработчике `activate`, уже после того как GTK сделал свой
`setlocale`) принудительно возвращает `LC_NUMERIC` в `"C"`, не трогая
остальные категории локали (переводы интерфейса и т.п. не страдают).

## Сборка

```bash
# Fedora, нативно
nim c -d:release --threads:on Monolit.nim

# Windows, кросс-компиляция с Fedora (mingw-w64)
nim c -d:release --threads:on --os:windows Monolit.nim
```

Первый запуск для каждой цели соберёт с нуля libvidstab → (libx264 на
Windows) → статический FFmpeg — это займёт время; повторные сборки
используют кэш в `vidstab_build*/`, `x264_build_windows/`,
`ffmpeg_build*/` (см. `config.nims`).

Перед сборкой под Windows на Fedora должен быть установлен рантайм GTK4
для mingw64 (`sudo dnf install mingw64-gtk4` либо через MSYS2), иначе
`pkg-config --libs gtk4` в `src/gtk4_api.nim` не найдёт нужные `.dll.a`.

Остальные зависимости (vid.stab, x264, FFmpeg) `config.nims` собирает
сам из исходников при первой сборке — вручную ставить их не нужно.

## Вкладки приложения

1. **Файл** — выбор входного видео, выбор пути сохранения результата,
   выбор контейнера результата (**mp4** либо **mkv** — определяет
   расширение файла и, соответственно, мьюксер libavformat; mkv надёжнее
   сохраняет субтитры и вложения, mp4 совместимее с проигрывателями и
   сайтами), независимые переключатели «копировать звук без
   перекодирования», «сохранить субтитры» и «сохранить вложения» (шрифты
   и т.п. в mkv) — все включены по умолчанию.
2. **Стабилизация**
   - Проход 1 (`vidstabdetect`): `shakiness` (1–10), `accuracy` (1–15),
     `stepsize`, `mincontrast` — по умолчанию выставлены на максимум
     точности анализа (10/15/6/0.3).
   - Проход 2 (`vidstabtransform`): `smoothing`, `zoom`, `optzoom`
     (0/1/2 — выкл/статичный/адаптивный), `crop` (`keep`/`black` —
     чёрные поля), `interpol` (по умолчанию `bicubic` — самая точная
     интерполяция), `maxshift`/`maxangle`.
3. **Резкость** — опциональные Smart Blur (шумоподавление ПЕРЕД
   резкостью), unsharp и Contrast Adaptive Sharpening (CAS); последние
   два применяются ПОСЛЕ стабилизации/зума (иначе усиливались бы
   артефакты исходной тряски, а не детали кадра) и их можно включить
   одновременно.
4. **Компрессия** — `preset` энкодера `libx264` и режим управления
   битрейтом:
   - **1 проход (CRF)** — постоянное качество на всём ролике; быстрее,
     но итоговый размер файла заранее не предсказать. Для наивысшего
     качества — `preset=slow`/`slower` и `CRF 16–18`.
   - **2 прохода (задать битрейт)** — `libx264` сначала проходит весь
     ролик, считая статистику сложности сцен (проход A), затем кодирует
     повторно (проход B), распределяя биты так, чтобы точно попасть в
     заданный целевой битрейт видео — вдвое медленнее, зато размер файла
     предсказуем. Ползунок «Целевой битрейт видео» активен только в этом
     режиме, а CRF — только в режиме 1 прохода; переключение режима
     сразу блокирует неиспользуемый контрол.

## Известные упрощения (что можно доработать дальше)

- Обрабатывается только первый найденный видеопоток
  (`av_find_best_stream`); мультивидеопотоковые контейнеры не
  поддерживаются.
- Аудио/субтитры копируются потоково «как есть»: если нужна конвертация
  формата аудио — это отдельная задача перекодирования, сейчас не
  реализована.
- Файл `transforms.trf` пишется во временную папку ОС и удаляется по
  завершении; отдельного UI для его сохранения/переиспользования между
  запусками нет.
